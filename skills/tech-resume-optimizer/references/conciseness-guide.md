# 内容精简指南 - 全量诊断模式

> **快速索引**: [核心原则](#核心原则) | [全量诊断](#全量诊断模式) | [逐条指导](#逐条指导策略) | [HTML报告优化](#html报告优化)

本文档提供简历优化的**全量诊断模式**策略，确保识别并列出简历中的每一处问题，为每处问题提供保姆级详细指导。

## 核心原则

### 1. 全量细致诊断（重要）

**目标**: 识别并列出简历中的每一处问题，严禁合并相似问题或仅展示代表性实例。

**关键要求**:

- **无遗漏覆盖**: 必须扫描整个简历，找出所有问题实例
- **逐条指导**: 即使存在10处"缺少量化数据"，也要为每一处单独提供改写建议
- **完整闭环**: 每个问题必须包含"原文定位 → 问题分析 → 具体改写 → 改写理由"
- **取消数量限制**: 移除对高/中优先级问题展示数量的限制，确保所有问题均被详细呈现

### 2. 强制生成HTML报告

**要求**: 每次简历优化任务必须生成完整的静态HTML优化报告，不得省略。

**原因**:

- HTML报告是核心输出，用户依赖其进行简历修改
- 可视化展示便于用户理解和执行修改建议
- 可保存、可打印、可分享，提升用户体验

### 3. 保姆级指导标准

**每个问题点必须包含**:

1. **原文定位**: 明确指出问题所在的具体位置和内容
2. **问题分析**: 解释为什么这是问题，影响是什么
3. **具体改写**: 提供可直接复制的改写版本
4. **改写理由**: 说明为什么这样改更好，引用行业标准或最佳实践

### 2. 篇幅控制标准

根据候选人经验级别,设定合理的简历篇幅:

| 经验级别     | 简历页数 | 总字数范围  | 项目经历数量         | 每项目描述长度 |
| ------------ | -------- | ----------- | -------------------- | -------------- |
| 初级(<2年)   | 1页      | 800-1200字  | 2-3个                | 150-250字/项目 |
| 中级(2-5年)  | 1-2页    | 1200-1800字 | 3-4个                | 200-300字/项目 |
| 高级(5-10年) | 2页      | 1800-2500字 | 4-5个                | 250-350字/项目 |
| 资深(>10年)  | 2页      | 2000-2800字 | 3-4个(聚焦最近5-8年) | 300-400字/项目 |

**重要提醒**:

- HR平均浏览简历时间仅**6-10秒**,必须在有限篇幅内突出核心价值
- 超过2页的简历会被认为"无法提炼重点"
- 项目描述过长会稀释亮点,每个项目控制在**3-5条要点**

## 全量诊断模式

### 问题识别策略

**全量扫描要求**:

- 对每类问题(如量化缺失、STAR不完整)，必须扫描整个简历找出**所有实例**
- 识别问题总数，并在报告中明确标注
- 严禁合并相似问题，每处问题都要单独展示

### 逐条指导标准

对于每个发现的问题，必须提供完整的四步指导：

#### 步骤1: 原文定位

```markdown
**原文**:

> 引用简历中的原始文本(标注所在章节，如"[工作经历: XX公司]"或"[项目: YY项目]")
```

#### 步骤2: 问题分析

```markdown
**问题分析**:

- 指出具体问题类型(如"缺少量化数据"、"STAR不完整")
- 说明问题的影响(如"无法证明实际能力"、"降低可信度")
- 标注严重程度(🔴 严重 / 🟠 高优先级 / 🟡 中优先级)
```

#### 步骤3: 具体改写

```markdown
**改写建议**:
提供具体的改写版本，优先使用STAR法则重构(参考 resume-best-practices.md 的三层结构:句式模板+词汇替换+完整示例)
```

#### 步骤4: 改写理由

```markdown
**改写理由**:
解释为什么这样改更好，引用行业标准或最佳实践
```

### 示例：全量诊断模式应用

**场景**: 简历中存在8处"缺少量化数据"的问题

**❌ 错误做法**(优先级筛选):

```markdown
### 问题类型 #1: 缺少量化数据 (共8个实例)

**严重程度**: 🔴 严重  
**展示策略**: 展示3个最具代表性实例，其余5处请参照示例统一修正

**实例 #1**: ...
**实例 #2**: ...
**实例 #3**: ...

**其他5处**: 请参照上述示例统一修正
```

**✅ 正确做法**(全量诊断):

```markdown
### 问题类型 #1: 缺少量化数据 (共8个实例)

**严重程度**: 🔴 严重  
**展示策略**: 全量展示所有8个实例，每处单独提供改写建议

---

#### 实例 #1: [工作经历: XX公司]

**原文**:

> "优化了页面加载速度"

**问题分析**:

- 问题类型: 缺少量化数据
- 影响: 无法证明优化效果，缺乏说服力
- 严重程度: 🔴 严重

**改写建议**:

> "通过代码分割和懒加载，首屏加载时间从**3.2秒**降至**1.5秒**(**降低53%**)，Lighthouse性能评分从**62**提升至**89**"

**改写理由**:
量化数据是简历的灵魂。根据统计，包含量化成果的简历获得面试邀请的概率提升**40%**。即使无法获取精确数据，也应提供合理估算。

---

#### 实例 #2: [项目: YY项目]

**原文**:

> "提升了知识召回准确率"

**问题分析**:

- 问题类型: 缺少量化数据
- 影响: 无法评估改进幅度，难以证明技术价值
- 严重程度: 🔴 严重

**改写建议**:

> "知识召回准确率从68%提升至**87%**，平均响应时间**<500ms**"

**改写理由**:
量化指标能够直观展示技术成果，帮助面试官快速评估你的贡献价值。

---

...(继续展示实例 #3 至 #8，每处都提供完整的四步指导)
```

### 禁止行为

**严禁以下做法**:

- ❌ 合并相似问题(如"全文共发现8处技术术语错误，请参考示例统一修正")
- ❌ 仅展示代表性实例(如"最多展示3个实例")
- ❌ 省略任何问题(即使问题很小，也要列出)
- ❌ 使用模糊指引(如"其他类似问题请参照上述示例")

**必须做到**:

- ✅ 每处问题单独展示
- ✅ 每处问题提供完整四步指导
- ✅ 明确标注问题总数
- ✅ 确保用户可以无需额外思考即可直接修改

## 逐条指导策略

### 指导深度要求

为确保用户能够直接执行修改，每条建议必须达到以下深度：

1. **精准定位**: 明确指出问题所在的具体位置和内容(标注章节名称)
2. **对比展示**: 原文 vs 修改建议，一目了然
3. **原理说明**: 解释为什么要这样改，引用行业标准或最佳实践
4. **量化预期**: 如果可能，说明修改后的预期效果(如"包含量化成果的简历获得面试邀请的概率提升40%")
5. **可操作性**: 提供具体可执行的修改步骤，而非空泛建议
6. **可直接复制**: 改写建议应该是用户可以直接复制到简历中的完整文本

### 示例模板

```markdown
#### 实例 #N: [章节名称]

**原文**:

> "引用简历中的原始文本"

**问题分析**:

- 问题类型: [具体问题类型]
- 影响: [说明问题的负面影响]
- 严重程度: 🔴 严重 / 🟠 高优先级 / 🟡 中优先级

**改写建议**:

> "提供具体的改写版本，可以直接复制到简历中"

**改写理由**:
解释为什么这样改更好，引用行业标准或最佳实践。例如：量化数据是简历的灵魂，根据统计，包含量化成果的简历获得面试邀请的概率提升40%。
```

### 常见错误避免

**❌ 错误做法**:

```markdown
**问题**: 缺少量化数据
**建议**: 添加具体数据
```

**问题**: 过于笼统，用户不知道如何添加数据

**✅ 正确做法**:

```markdown
**原文**:

> "优化了页面加载速度"

**问题分析**:

- 问题类型: 缺少量化数据
- 影响: 无法证明优化效果，缺乏说服力
- 严重程度: 🔴 严重

**改写建议**:

> "通过代码分割和懒加载，首屏加载时间从**3.2秒**降至**1.5秒**(**降低53%**)，Lighthouse性能评分从**62**提升至**89**"

**改写理由**:
量化数据是简历的灵魂。根据统计，包含量化成果的简历获得面试邀请的概率提升**40%**。即使无法获取精确数据，也应提供合理估算(如"约"、"超过"、"低于")。
```

**优势**: 用户可以直接复制改写建议，无需额外思考

## HTML报告优化

### 报告结构要求

**必需章节**(始终展示):

1. 总体评分(4维度)
2. 优势与改进点(各5-7条)
3. **详细问题分析**(全量展示，按问题类型分组，每组包含所有实例)
4. 技术栈深度分析
5. 模拟面试问题
6. 面试准备建议
7. 立即执行的行动计划
8. 总结与建议

### 问题分组展示优化

**当前要求**: 全量展示所有问题实例，不得省略

**实现方案**:

1. **按问题类型分组**: 将同类问题(如"缺少量化数据")归为一组
2. **组内全量展示**: 每组内展示该类型的所有实例，不得省略
3. **紧凑布局**: 使用紧凑的卡片设计，减小实例间距
4. **视觉层次**: 通过颜色和大小区分严重程度，让用户快速定位重点

**示例HTML结构**:

```html
<!-- 问题类型组 -->
<section class="problem-group">
  <h3>问题类型 #1: 缺少量化数据 (共8个实例)</h3>
  <p class="severity severe">严重程度: 🔴 严重</p>

  <!-- 实例 #1 -->
  <article class="instance-card">
    <div class="instance-header">
      <span class="instance-number">#1</span>
      <span class="instance-location">[工作经历: XX公司]</span>
    </div>
    <div class="instance-content">
      <div class="original-text">
        <strong>原文:</strong>
        <blockquote>"优化了页面加载速度"</blockquote>
      </div>
      <div class="analysis">
        <strong>问题分析:</strong>
        <ul>
          <li>问题类型: 缺少量化数据</li>
          <li>影响: 无法证明优化效果，缺乏说服力</li>
          <li>严重程度: 🔴 严重</li>
        </ul>
      </div>
      <div class="suggestion">
        <strong>改写建议:</strong>
        <blockquote>
          "通过代码分割和懒加载，首屏加载时间从<strong>3.2秒</strong>降至<strong>1.5秒</strong>(<strong>降低53%</strong>)，Lighthouse性能评分从<strong>62</strong>提升至<strong>89</strong>"
        </blockquote>
      </div>
      <div class="reason">
        <strong>改写理由:</strong>
        <p>量化数据是简历的灵魂。根据统计，包含量化成果的简历获得面试邀请的概率提升<strong>40%</strong>。</p>
      </div>
    </div>
  </article>

  <!-- 实例 #2 -->
  <article class="instance-card">
    <!-- 同样完整的四步指导 -->
  </article>

  <!-- ... 继续展示所有8个实例 -->
</section>
```

### 篇幅控制建议

虽然要求全量展示，但仍需注意报告的可读性：

- **紧凑布局**: 减小卡片间距，使用更紧凑的设计
- **清晰层次**: 通过颜色、大小、间距创造清晰的视觉层次
- **导航辅助**: 添加目录或锚点，方便用户快速跳转
- **打印优化**: 确保打印时格式正常，分页合理

**注意**: 不再限制HTML报告总长度为50KB，应以完整展示所有问题为首要目标。

## 常见陷阱避免

### ❌ 陷阱1: 过度追求全面

**表现**: 试图找出并展示所有问题,报告长达20+页

**后果**:

- 用户阅读疲劳,放弃查看
- 重点不突出,不知道先改哪个
- 修改工作量巨大,产生畏难情绪

**解决**:

- 严格执行优先级筛选规则
- 聚焦"修正后收益最大"的问题
- 提供分阶段改进计划(本周/本月/长期)

### ❌ 陷阱2: 忽略用户承受能力

**表现**: 一次性提出20+修改建议

**后果**: 用户感到压力过大,可能放弃优化

**解决**:

- 高优先级建议控制在5-8条
- 提供"最小可行改进"方案(只改最关键的3条)
- 明确告知"完成这些即可投递,其余可后续迭代"

### ❌ 陷阱3: 建议过于理论化

**表现**: 给出大量原则性建议,缺少具体操作

**后果**: 用户知道要改,但不知道怎么改

**解决**:

- 每条建议必须包含"原文 → 修改建议 → 理由"
- 提供可直接复制的改写模板
- 标注预计耗时,便于用户规划

- [ ] **全量扫描**: 已识别简历中的所有问题实例
- [ ] **逐条指导**: 每个问题都提供了完整的四步指导(原文→分析→改写→理由)
- [ ] **无遗漏**: 没有合并相似问题，每处问题都单独展示
- [ ] **强制HTML**: 已生成完整的静态HTML优化报告
- [ ] **HTML全量展示**: HTML报告中包含了所有发现的问题
- [ ] **可直接执行**: 所有建议都有具体可操作的改写示例，用户可以直接复制
- [ ] **视觉清晰**: HTML报告视觉层次清晰，关键信息突出
- [ ] **无占位符**: 检查无占位符残留(如"[姓名]"、"[公司名称]")
